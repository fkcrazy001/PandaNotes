# 性能调优方法论

DPDK 调优最怕两种做法：一种是只盯着某个“神秘参数”；另一种是看到官方推荐就全开一遍，最后自己也说不清性能到底为什么变好或变坏。

我更倾向于把调优拆成一条很朴素的链路：**先确认瓶颈在哪一层，再调那一层。** 不然很多优化动作只是碰巧有效，换台机器就失灵。

---

## 先分清三类瓶颈

大多数 DPDK 场景，瓶颈通常落在这三类之一：

- CPU 算力不够
- 内存/NUMA/cache 行为不好
- 设备/队列/offload 没配对

如果这三类都没分清，就开始调 burst、调 cache、调阈值，基本等于蒙。

---

## 第一层：CPU 与核拓扑

先确认下面几件事：

- worker 核有没有和业务主循环一一对应
- RX/TX queue 是否和 lcore 绑定清楚
- service core / 后台线程有没有抢业务核
- 是否跨 socket 跑了本来可以本地完成的路径

这一步里最常见的问题，不是 SIMD 没开，而是核用乱了。

尤其是：

- 多核共享一个 RX queue
- 某个 worker 顺手还跑了后台任务
- port 在 socket 0，mempool 在 socket 1，worker 又在 socket 1 或 0 乱跳

这些都会直接把时延和吞吐打坏。

---

## 第二层：内存与对象生命周期

DPDK 性能很多时候不是算力瓶颈，而是“每个包都在跨 NUMA、抢 cache line、等对象池”。

重点看：

- mempool 在哪个 socket
- mbuf object size 是否过大
- mempool cache size 是否合理
- hugepage 模式是否适合当前部署

如果 RX refill 或 TX recycle 已经开始拖慢主循环，那再怎么抠 PMD 指令级优化，收益都有限。

---

## 第三层：burst 粒度

DPDK 很多性能都建立在 burst 语义上。burst 太小，会导致：

- 函数调用成本摊不薄
- 预取效果差
- 向量化路径起不来
- ring/mempool 的批量优势发挥不出来

但 burst 也不是越大越好。太大时常见问题是：

- 增加单轮处理尾延迟
- 在流量不稳时更容易造成抖动
- 某些场景下 cache 污染更明显

所以 burst 更像“吞吐/时延”的旋钮，不是单向越大越优。

---

## 第四层：queue 与 descriptor 参数

调 queue 时，至少要看：

- RX/TX queue 个数
- ring size
- `tx_free_thresh`
- `tx_rs_thresh`
- PMD 的预取/写回阈值

特别是 TX 侧，很多“mempool 不够”“发包抖动”问题，本质都和回收阈值、descriptor 回写节奏有关。

这类参数不应该离开 workload 单独谈。小包转发、大包封装、虚拟化场景、纯 L2 转发，最优点往往不一样。

---

## 第五层：offload 与向量化

不少优化不是你“调用了 API 就有”，而是要满足条件 PMD 才会走最优路径：

- checksum / TSO / RSS / VLAN 等 offload 是否匹配硬件能力
- queue/ring 配置是否满足 vector path 条件
- mbuf 布局和数据对齐是否干净

这意味着调优不能只看源码里“有 vector 实现”，还要确认运行时是否真的命中了它。

---

## 第六层：规则与分类路径

如果应用里有 `rte_flow`、ACL、hash、LPM 这类分类逻辑，调优时一定要区分：

- 分类发生在硬件还是软件
- 命中代价是不是已经高于收发本身
- rule update 是否干扰热路径

很多系统前期只看 RX/TX PPS，后面功能变复杂后才发现瓶颈已经转到分类器或规则管理上了。

---

## 指标怎么选

我更建议最少看这几类指标：

- PPS / 吞吐
- 单核 cycles per packet
- queue level drops
- xstats 里的异常计数
- cache miss / NUMA 远端访问
- mbuf pool 可用量与回收节奏

只看最终吞吐很危险，因为很多“吞吐差不多”的方案，尾延迟、CPU 利用率和稳定性差别会非常大。

---

## 调优顺序建议

我的习惯顺序通常是：

1. 先固定 workload 和流量模型
2. 先看核绑定、NUMA、queue 拓扑
3. 再看 mempool/mbuf 生命周期
4. 再调 burst 和 descriptor 阈值
5. 最后才看向量化、offload、细碎参数

这个顺序的原因很简单：越往前的层，越容易形成数量级影响；越往后的层，越像精修。

---

## 常见误区

### 1. 一上来就调 PMD 私有参数

但实际上 queue 绑定和 NUMA 都还没对齐。

### 2. 用一个指标统治一切

例如只看 PPS，不看 drops、CPU 占用和尾延迟。

### 3. 用 testpmd 参数直接套业务程序

testpmd 很适合摸硬件边界，但业务程序的对象生命周期、规则路径、内存访问完全可能不一样。

### 4. 调参不隔离变量

一次动五个参数，最后只能得到“好像快了”，得不到原因。

---

## 一个更实用的总结

DPDK 调优不是找某个万能参数，而是在回答一个更基础的问题：

- 包在哪一层最耗 CPU
- 对象在哪一层最耗内存带宽
- 数据在哪一层最容易跨 NUMA 或发生回压

一旦把瓶颈定位清楚，很多参数其实都不神秘。它们只是对应某一层资源分配方式的旋钮而已。
